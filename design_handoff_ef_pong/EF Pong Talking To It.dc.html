How the app talks to it

The database is storage; it doesn't do anything on its own. The API is the fixed menu of things the app is allowed to ask for. This step defines that menu — the complete list is short — and settles the one question that guards fairness: where ELO gets calculated.

1What an API actually is

An API is a menu of allowed operations. The frontend never reaches into the database directly; it can only order items off the menu: “list the leaderboard,” “log this match,” “post this comment.” The API takes the order, does the work against the database, and hands back the result.

Two payoffs from this one idea:

  • Safety. The database password lives only inside the API, never on a phone. A phone can only do what the menu allows — it can't wipe a table or hand itself 500 ELO.
  • One set of rules. “A game is to 11, win by 2,” “ELO is zero-sum” — enforced once, in the API, so every phone and the wall always agree.
You mostly get this for free. From Step 2: Supabase auto-generates a secure API over your seven tables the moment you define them — the simple “read this / add that” items appear on the menu automatically. You only hand-write the one operation that's more than a plain read/write: logging a match (§3).

3Why ELO must run on the server — not the phone

The prototype computes ELO in the browser (calcDelta) — correct for a demo, wrong for a shared app. Move that exact math to the API, and here's what it buys:

  • Everyone agrees. One implementation of the formula, run in one place. No chance two phones disagree about a rating.
  • It can't be faked. A phone asks to log “I beat Marco 11–0”; the server decides the points. Nobody can hand themselves ELO by editing the app.
  • Zero-sum stays intact. A match touches three tables (match + both standings + history). The server does them as one atomic unit — all succeed or none do — so two people logging at once can never half-apply and corrupt ratings. This is the concurrent-write problem from Step 2, solved.
Decision: ELO is computed server-side, inside logMatch. The frontend sends only the raw facts — who played, the score — and displays whatever the server returns. The prototype's expected() / marginMult() / calcDelta() move verbatim; the numbers don't change, only where they run does.

4What logMatch does, in order

This is the one operation worth seeing spelled out — on Supabase it's a small server-side function (a “database function”) the app calls by name. In plain steps:

1Receive winner, loser, scores from the app — nothing else is trusted
2Validate: to 11, win by 2, two different players — reject if not
3Read both players' current standings for the active season
4Compute elo_delta — expected × margin × K, the prototype's math
5Write the match row (delta frozen in, status = confirmed)
6Update standings: winner +delta & +1 win, loser −delta & +1 loss
7Append two rating_history rows — the audit trail
8Return the result so the app shows “+22 ELO” — all of 5–7 as one unit

Every other operation is a simpler version of this shape. Once logMatch makes sense, the API is done.
