Backend walkthrough — Step 1 of 7: the data

The data model, drawn out

These are the PRD's tables, but concrete: every column, its type in plain terms, and the exact arrows between tables. This is the shape your database will physically take — and it already matches the players and matches shapes in the prototype, so nothing about the frontend has to change. There are seven tables: the PRD's six, plus standings (§4), which we added to handle the quarterly reset cleanly.

1How to read this

Each box below is one table (one spreadsheet). Each line is a column: its name, its type, and what it holds. Two badges mark the special columns:

PK  Primary key — the row's unique ID; nothing else in the table repeats it. FK  Foreign key — holds the ID of a row in another table (an arrow).
Types, in plain terms: text words int whole number bool true/false timestamp a date & time enum one of a fixed set of words
One rule worth knowing: a foreign key can only point at an ID that actually exists. The database refuses to save a match whose winner isn't a real player — so you can never end up with a match pointing at nobody. That guarantee is a big part of why we store data this way instead of in loose JSON.

2The seven tables

playerone row per person on the roster (~40)
id PKtextUnique handle, e.g. marco. Everything else points here.
nametextDisplay name — “Marco Rossi”.
initialstextAvatar text — “MR”. (Could be computed from name instead.)
avatar_colortextHex for the avatar chip, e.g. #006BD6.
join_datetimestampWhen they were added. Set automatically.
login_identitytextEmpty for now. Later holds the verified email / EF SSO id (Step 5). Adding it later changes nothing above.

Notice what's not here: current ELO, wins, losses, rank. Those change every season, so they live in standings below — one row per player per season. Everything else (rank order, head-to-head, streaks) is computed on demand, never stored.

standingsone row per player per season — the leaderboard, cached
id PKtextUnique ID.
player_id FKtext→ player.id.
season_id FKtext→ season.id. The pair (player + season) is unique — one standing per person per quarter.
elointCurrent rating this season. Starts at 1000; the server updates it on every match.
winsintWins this season.
lossesintLosses this season.
peakintHighest ELO reached this season.

This is the exact set of numbers the prototype shows on each player — just moved to where they belong. Past seasons' rows stay frozen, which is the Hall of Fame. See §4 for why this beats recomputing every read.

matchone row per game played — the heart of the app
id PKtextUnique match ID, generated on save.
winner_id FKtext→ player.id of the winner.
loser_id FKtext→ player.id of the loser.
winner_scoreinte.g. 11.
loser_scoreinte.g. 4.
elo_deltaintPoints transferred (winner +, loser −). Computed by the server (Step 3) and frozen here so the feed always shows what really happened.
season_id FKtext→ season.id this game counted toward.
entered_by FKtext→ player.id who logged it. (Powers “who to trust” once login lands.)
statusenumOne of confirmed / pending / disputed. Launch: always confirmed. This one column is what makes Option B a config change, not a rebuild.
played_attimestampWhen it happened. Set automatically; sorts the feed.
rating_historyone row per player per match — the audit trail
id PKtextUnique ID.
player_id FKtext→ player.id.
match_id FKtext→ match.id that caused this change.

This table is append-only history: a new row every time anyone's rating moves. It's also your safety net for admin corrections (Step 6) — you can replay it.

3The arrows — how the tables connect

Read → as “points at one,” and many as “there can be lots of these.” This is the whole relational structure of EF Pong on one screen:

many match → player   (winner, loser, entered_by — three arrows into player)
many match → season
many rating_history → player  and  → match
many reaction → match
many comment → match  and  → player (author)
many standings → player  and  → season (unique per pair)
one season → player (champion)

player and season are the two “anchor” tables everything else refers back to. If you can picture a match holding two player-IDs and a season-ID, you've understood the model.

4Why standings exists Decided

The prototype keeps elo, wins, losses right on each player. That's perfect for a single browser — but ratings reset every quarter, and the Hall of Fame needs the old numbers kept. So “current ELO” isn't really a fact about a player; it's a fact about a player in a season. Two honest ways to handle it — we chose B:

ApproachHowTrade-off
A. Recompute every readStore nothing; replay all of this season's matches to get everyone's ELO whenever the leaderboard loads.Always correct, zero duplication — but wasteful, and slower as matches pile up. Fine at 40 players, sloppy in principle.
B. A standings table ChosenThe table in §2: a row per player×season holding current elo, wins, losses, peak. The server updates it when a match is saved.One fast read for the leaderboard; past seasons stay frozen for the Hall of Fame automatically. The rating_history table keeps it honest if it ever drifts.
Decision: standings is a first-class table (§2). It's the clean answer to the seasonal reset, it makes the leaderboard a single quick query, and it's exactly the numbers the prototype already shows — just moved to where they belong. This is the only addition to the PRD's original table list, and it's additive: the frontend is unaffected.
