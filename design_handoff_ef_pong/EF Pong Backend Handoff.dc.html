EF Pong — backend & database walkthrough

This brief tracks the backend walkthrough for EF Pong. It summarises what the product is, what's already built, every decision we've locked (all 7 steps done), the data we store, and — now that design is complete — how to hand this to Claude Code and what to build next.

1What EF Pong is

A live, always-on leaderboard for office ping pong (~40 players at EF). After a game, a player logs the result (e.g. “I beat Sofia 11–4”); the app updates each player's ELO rating, re-ranks the ladder, and posts the match to a public feed where colleagues react and talk trash. Quarterly seasons reset the ladder; past champions live in a Hall of Fame. Used on phones (to log) and on an office wall display (to watch).

2What already exists (in this project)

Product & prototype:

  • EF Pong PRD.dc.html — the full product requirements (v0.2, decisions locked; kept as source of truth).
  • EF Pong Prototype.dc.html — a working, clickable frontend prototype: leaderboard, log-a-match with live ELO, feed with reactions + comments, profile + head-to-head, wall display, and a “pick-your-name” identity gate. It uses seeded in-memory data — nothing is saved; there is no backend yet. The design is done; making it deployable and wiring it to the backend is build work still ahead.

Backend design (this walkthrough, in order):

  • EF Pong Data Model.dc.html — Step 1: the seven tables drawn concretely (columns, types, keys, relationships).
  • EF Pong Where Data Lives.dc.html — Step 2: the database platform decision (Supabase).
  • EF Pong Talking To It.dc.html — Step 3: the API operations and server-side ELO.
  • EF Pong Hosting And Cost.dc.html — Step 4: where the frontend and backend run, and the cost.
  • EF Pong The Auth Path.dc.html — Step 5: the login road — pick-your-name → magic link → EF SSO.
  • EF Pong Admin And Corrections.dc.html — Step 6: voiding/fixing a wrong match and recomputing ratings (correctMatch).
  • EF Pong The Live Wall.dc.html — Step 7: keeping the office display live (Supabase realtime, polling fallback).

Everything is built on the EF Backpack design system bound to this project. Keep using it for any new UI.

4The data we need to store

A database is just a set of linked spreadsheets (“tables”); each has columns and rows, and rows reference each other by ID. These are the tables EF Pong needs (everything else — rankings, head-to-head, upsets — is computed from them). The full data model, with every column and relationship, is drawn out in EF Pong Data Model.dc.html:

TableColumns (roughly)
Playerid, name, avatar, join date, (later) login identity
Standingsid, player→Player, season→Season, current ELO, wins, losses, peak — one row per player per season (the cached leaderboard; past seasons frozen = Hall of Fame)
Matchid, winner→Player, loser→Player, winner score, loser score, ELO change, timestamp, season, entered-by, status (confirmed/pending/disputed)
RatingHistoryid, player→Player, match→Match, rating after, timestamp
Seasonid, name, start, end, champion→Player, active flag
Reactionid, match→Match, player→Player (or anonymous), type
Commentid, match→Match, author→Player, text, timestamp

5Progress — done and what's left

The backend walkthrough is a 7-step plan: the PRD's open questions, with the data model added up front. All seven steps are now done and their decisions are written into the docs above. The approach was design first (finish all seven), then build — the spec is settled, so building can begin.

After Step 7, the spec is complete and build begins: create the Supabase project → recreate the seven tables → seed ~40 real players → point the frontend's data layer at Supabase (shapes already match) → move ELO to the server → add realtime for the wall. An optional early checkpoint: get just players reading live end-to-end to prove the setup.

6Next: handoff to Claude Code — the build

Design is done end-to-end. The next phase is implementation, handed to a developer using Claude Code. A ready-to-download handoff package has been prepared — design_handoff_ef_pong/ — containing a self-sufficient README.md (the full spec: data model, the ten API operations, the ELO parameters, corrections, auth phases, realtime, tokens, and build order) plus copies of the prototype and all nine design docs and the design-system files needed to open them.

What Claude Code should do, in order

  1. Set up Supabase. Create the project; recreate the seven tables from the Data Model doc (columns, types, keys, relationships).
  2. Open the first season. No player seeding — players self-register via the in-app create-player flow (each starts at 1000 ELO); the roster fills in live from day one.
  3. Prove the path. Get just players reading live end-to-end — confirms the GitHub Pages ↔ Supabase wiring in one small step.
  4. Wire the frontend. Recreate the prototype in a deployable codebase and swap its in-memory seeds for the ten API operations (shapes already match the tables by design — a swap, not a rebuild).
  5. Move ELO server-side. Port calcDelta verbatim into the logMatch database function (start 1000, K=24 / 40 provisional, margin multiplier, zero-sum, floor ~100); add rollSeason and correctMatch (void/edit + full-season replay).
  6. Go live. Turn on Supabase realtime for the wall (polling fallback); publish the frontend to GitHub Pages.
  7. Later, tighten trust. Phase 2 auth — enable email magic-link, fill login_identity, switch matches to confirm-or-dispute. EF SSO (Phase 3) much later.

Deliverable: download design_handoff_ef_pong/ as a zip and open it in Claude Code. The README.md is written to stand alone — a developer who wasn't in these sessions can build from it. Keep the four rules above the build steps intact: server-side ELO, atomic writes, standings is a rebuildable cache, and the auth/status hooks (login_identity, match.status) stay in from day one even while unused.
