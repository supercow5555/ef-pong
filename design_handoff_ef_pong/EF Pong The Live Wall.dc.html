Backend walkthrough — Step 7 of 7: the live wall

Keeping the wall live

The office display has to stay current on its own — nobody walks over to tap refresh. There are two honest ways to keep a screen fresh: it can keep asking for news, or the server can tell it the moment something changes. This last step picks between them and settles what “live” actually costs to build.

1The two ways a screen stays fresh

Polling

Keep asking

The wall re-runs getLeaderboard and getFeed every few seconds and redraws if anything changed.

+ Dead simple, works everywhere, self-healing.
− A few seconds of lag; constant requests (trivial at our size).

Push (realtime)

Get told

The wall subscribes once; the server pushes a nudge the instant a match, reaction or comment is written, and the wall refreshes.

+ Instant, efficient, feels alive.
− A touch more setup than a timer.

2Why push is basically free here

Normally push is the fancier, harder option — but from Step 2, Supabase ships realtime in the same box as the database. You can subscribe to a table (“tell me whenever match changes”) and it pushes the change over a live connection. So the hard part of push is already built; you just switch it on and listen.

Decision: use Supabase realtime for the wall, keep polling as the fallback. Realtime is included, it's the nicer experience (a logged match pops onto the wall and the ELO animates within a second), and it's little more than “subscribe to the tables, refetch on change.” If realtime ever misbehaves, a 10-second poll of the same two operations is a couple of lines and an instant, no-drama fallback.

3What actually updates — and how little code it is

The wall already knows how to draw itself from getLeaderboard and getFeed (Step 3). “Live” just means re-running those when the data behind them moves — the ranking re-orders, the ticker gains a match, biggest-mover and the champion update. Same reads, retriggered.

1On load, the wall calls getLeaderboard + getFeed once and draws
2It subscribes to changes on match, standings, reaction, comment
3Any change → refetch the affected view & animate the diff — new row slides in, ELO ticks
4Fallback: if the connection drops, re-poll every 10s until it's back — self-healing

4Staying up for days unattended

The PRD asks the wall to run for days without babysitting. Both approaches handle it, and it's worth being deliberate:

  • Auto-reconnect. Supabase's realtime client reconnects on its own after a network blip; a poller is inherently self-healing (the next tick just works). Either way, on reconnect the wall does a full refetch so it can never sit on stale data.
  • Refresh on wake. If the screen or tab sleeps, refetch everything when it wakes — treat waking like a fresh load.
  • No memory creep. The view redraws from fresh reads rather than piling up state, so a screen left on all quarter stays healthy.

5That's the whole backend spec

Seven steps, done. The design is settled end to end: the seven tables, on Supabase, behind a ten-operation API with server-side ELO, with the frontend on GitHub Pages, a phased auth path, safe admin corrections, and a live wall. No open questions remain — time to build.

Build order, once you're ready

  1. Create the Supabase project → recreate the seven tables from Step 1 in its table editor.
  2. Seed ~40 real players and open the first season.
  3. Point the prototype's data layer at Supabase (shapes already match) — swap in-memory for API calls.
  4. Move the ELO math into the logMatch database function; add rollSeason and correctMatch.
  5. Turn on realtime for the wall; publish the frontend to GitHub Pages.

An easy first checkpoint: get just players reading live end-to-end — it proves the whole GitHub Pages ↔ Supabase path in one small, satisfying step.
