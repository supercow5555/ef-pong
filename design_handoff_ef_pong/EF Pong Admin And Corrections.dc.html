Fixing a wrong match, safely

Someone will fat-finger a score, log the winner as the loser, or pick the wrong opponent. This step defines correctMatch — how an admin voids or fixes one match and puts every rating it touched back the way it should be. The hard part isn't the edit; it's undoing the ripple the bad match sent through the ladder.

1Why you can't just delete the row

ELO is zero-sum and chained: every match reads the two players' current ratings and moves them. So a wrong match doesn't just record one bad result — it feeds a wrong starting number into every match those players played afterward. Quietly deleting the bad row leaves all that downstream damage in place.

M1  ok→ M2  WRONG→ M3→ M4→ M5

Fix M2 and every downstream match needs its numbers recomputed too, because each was calculated from ratings the wrong match had already nudged.

2The safety net you already built

Correcting is only possible because two things from Step 1 are true: every match is stored in order with its raw facts (who played, the score), and rating_history is an append-only log of every rating change. Together they mean the season can be replayed from scratch and will land on exactly the same numbers — the ELO formula is deterministic (Step 3). Nothing is lost; nothing is guessed.

3The fix: void or edit, then replay

Every correction is the same two moves — change the one bad match, then rebuild the ratings that depended on it.

SituationMove 1 — the editMove 2 — the rebuild
Match never happened (double-logged, fake)Void it — mark the row voided so it no longer counts.Replay the season, then rewrite standings + rating_history.
Wrong details (score, or winner/loser swapped)Edit the match's facts to the truth.

How “replay” works — and the simplest version to ship

You could recompute only from the corrected match forward. Simpler and just as fast at our size: replay the entire active season from scratch. Reset everyone to 1000, run every non-voided match of the season back through the same logMatch math in time order, and overwrite standings and rating_history with the results. A season is at most a few hundred matches, so this is instant — and it's the version that's obviously correct, which is what you want for anything touching ratings.

Decision: correctMatch = void/edit the match, then full-season replay. One code path covers both void and edit, it reuses the ELO you already trust, and “recompute the whole season” is nearly impossible to get subtly wrong. Optimise to a partial replay only if a season ever grows big enough to notice — unlikely at 40 players.

4What correctMatch does, in order

Same shape as logMatch (Step 3, §4) — a server-side function, all-or-nothing:

1Admin picks the match and chooses void or edit — with the corrected facts
2Apply the change to that one match row
3Reset the season's standings to 1000 and clear its rating_history
4Replay every non-voided match of the season in time order, recomputing each delta
5Rewrite standings and re-append rating_history from the replay
6Done as one unit — the feed & leaderboard now show the corrected reality

Because steps 2–5 succeed or fail together, nobody ever sees a half-corrected ladder — the same atomic guarantee that protects logMatch.

5Who's allowed to do it

Corrections are powerful, so they're locked to a single admin — the product owner — matching the moderation decision in the PRD. How that's enforced tracks the auth path:

PhaseWho can run correctMatch & rollSeason
Launch (pick-your-name)No public admin UI. The owner runs corrections directly in the Supabase dashboard, or via a hidden owner-only action. Rare enough that this is fine.
After login (Phase 2+)An is_admin flag on the owner's player/identity; the server rejects the operation for anyone else. Same door for deleting comments.

6Why this keeps the data honest

  • Nothing is silently lost. A void is a marked state, not a hard delete; the original match still exists, just excluded from the replay. You can always see what was corrected.
  • Auditable. Because the fix is “replay from the stored facts,” the corrected ratings are reproducible — and you can record who corrected what if disputes ever need it.
  • Self-healing. If standings (the cache) ever drifts for any reason, the exact same replay rebuilds it from the matches. Corrections and cache-repair are one mechanism.
Guardrail: keep corrections admin-only and rare. They rewrite history for good reasons (a genuine mistake), so the bar to run one should stay high — it's a fix tool, not an undo button for a loss you didn't like.
