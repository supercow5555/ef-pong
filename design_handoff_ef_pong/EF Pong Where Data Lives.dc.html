Where the data lives

You have the seven tables (Step 1). Now: what real thing do they live inside, and how do you get one without becoming a database administrator? This is the highest-leverage call in the whole backend — it decides how much you have to build yourself.

1The four layers you need — and who runs each

Any app like EF Pong is really four layers stacked up. The only question that matters is which layers you run yourself vs. which a service runs for you:

FrontendThe app you already built. Runs in each phone / the wall browser. You have this.
APIThe rule-keeper. The one program that holds the database password and enforces “recompute ELO on save.” (Step 3.)
DatabaseThe storage. Your seven tables, on a disk, kept safe and backed up.
HostingThe always-on computer the API and database run on, reachable from the office. (Step 4.)

You could assemble all four by hand. As a first-timer you shouldn't — there's a category of service that hands you the bottom three, pre-wired.

2The two honest paths

Path A — Rent a bare database, build the API yourself

You sign up for a hosted database (e.g. a Postgres box from Amazon RDS, or a smaller host), then write and deploy a separate API program that talks to it. Four layers, you wire three.

You learnSQL, a backend language/framework, deployment, connection security, backups.
Good whenyou have unusual needs or existing backend skill. Neither applies here yet.
For EF PongOverkill. Weeks of plumbing before the first match is saved.

Path B — A backend-as-a-service (BaaS) Decided

One signup gives you the database and an auto-generated API over it and the hosting, already connected. You define your seven tables in a web dashboard; it instantly gives you secure ways to read and write them. Login, realtime updates, and file storage come in the same box — which is exactly your Steps 5 and 7.

You learnhow to define tables in a dashboard, and how to call a ready-made API. That's most of it.
Real optionsSupabase (a managed Postgres — a real SQL database, so Step 1 maps 1:1), or Firebase (document-style). Both have free tiers that comfortably cover 40 players.
For EF PongYou could have the seven tables live and saving real matches this week.
Decision: Path B on Supabase. It's a genuine Postgres database, so everything you designed in Step 1 — tables, foreign keys, the standings cache — drops in exactly as drawn. It also ships the three things still ahead of us for free: an auto-generated API (Step 3), built-in login incl. email magic-link and SSO (Step 5), and live realtime updates for the wall (Step 7). One tool retires four of your seven open questions.

3Why not just a shared Google Sheet or a JSON file?

Reasonable instinct at this size — here's where each breaks:

OptionWhere it breaks for EF Pong
Google SheetNo safe way to let 40 phones write at once, no server to compute ELO, no login story, no live push to the wall. Fine as a scoreboard, not as an app backend.
A JSON file on a serverTwo people logging a match at the same moment can overwrite each other — and with zero-sum ELO that silently corrupts ratings. Databases exist precisely to prevent this.
A real database (BaaS)Handles concurrent writes safely, enforces the foreign keys from Step 1, and grows with you. The right floor to start on.

4What picking Supabase actually commits you to

  • Cost: free tier covers ~40 players with room to spare; realistic worst case is a low monthly fee only if it gets popular. We'll confirm in Step 4 (Hosting).
  • Lock-in: low. Because it's standard Postgres underneath, your data and schema are portable if you ever outgrow it — you're not trapped.
  • What you do next: create a project, then recreate the seven tables from Step 1 in its table editor. That single act is setting up your database.
  • What you still design with me: the exact API operations (Step 3), where ELO runs, the login path (Step 5), corrections (Step 6), and the live wall (Step 7). Supabase provides the machinery; the rules are still yours.
