Where it runs, and what it costs

Supabase (Step 2) already runs three of the four layers for you. This step places the last one — the app itself — so every phone and the wall can reach it, shows where the prototype fits in that picture, and adds up the bill. Short version: two managed services, both free at this size, each just a web address.

1Only one layer is still homeless

Back in Step 2 we split the app into four layers. Supabase covers the bottom three — it stores your data (database), hands you the auto-generated API, and keeps both running on an always-on computer somewhere (backend hosting). That leaves exactly one thing to find a home for: the frontend — the app you already built.

The good news about the frontend: it's a “static” app — just files (HTML, JavaScript, a stylesheet) that run inside each person's browser. It needs no server of its own to think for it; the thinking happens on the phone and in Supabase. Static files are the cheapest, simplest thing on the internet to host.

2Where the frontend goes — a static host

A static host is a service that takes your app's files and serves them at a web address, worldwide, fast. You connect it to where your code lives, and every time you update the app it re-publishes automatically. You never touch a server.

Real optionsWhat you get
GitHub Pages ChosenIf the code lives in GitHub, publishing is one setting in the same place — no second account. Free, and it serves the files at yourorg.github.io/ef-pong.
VercelConnect the code, get a URL like efpong.vercel.app in minutes. Slightly slicker auto-publish and preview builds out of the box.
Netlify / Cloudflare PagesEffectively the same deal as Vercel — fine alternatives, all free.

All of them do the identical job here, and you can move between them later without changing a line of the app. Decision: GitHub Pages — the code will already be on GitHub, so it's the fewest moving parts and no extra sign-up.

One honest note on the URL: GitHub Pages doesn't get you a better free address than the others — its default is yourorg.github.io/ef-pong, just a different shape. A genuinely nice URL like efpong.ef.com needs a purchased custom domain, and that works identically on Pages, Vercel or Netlify (see §5).

3How the office reaches it — the whole picture

Both halves live on the public internet, each behind its own web address. An office phone opens the frontend's URL; the app files load into that phone's browser; from there the app talks directly to Supabase's URL for data. The wall display does exactly the same thing — it's just another browser open to the same address.

Worth noticing: the static host never talks to Supabase. It only delivers the app to the browser once; after that, the browser and Supabase talk to each other. That's why the frontend host can be so cheap — it does almost nothing.

4Where the prototype fits

The prototype you already have is the frontend — every screen, styled and clickable. The only difference between it and the deployed app is where its data comes from: today it invents data in memory and forgets it on refresh; the deployed version reads and writes the same shapes from Supabase instead.

  • It's not throwaway. The prototype is the deploy artifact-in-waiting. Its player and match shapes already match the tables (that was deliberate in Step 1).
  • Going live is a swap, not a rebuild. Replace the in-memory data layer with calls to Supabase's API (Step 3's menu), publish the files to the static host, done. The look and the flows don't change.

5The bill

At ~40 players this sits comfortably inside two free tiers. Realistically you pay nothing to run it.

€0 / moGitHub Pages — free static hosting. Far more traffic headroom than 40 people generate.
€0 / moSupabase — free tier covers the database size, API calls, auth and realtime for this scale with room to spare.
~€10 / yrOptional — a custom domain (e.g. efpong.ef.com) if you'd rather not use the free .vercel.app address. Purely cosmetic.
Realistic worst case: only if EF Pong got dramatically more popular than one office would Supabase's paid tier (around €25/mo) come into play — and even that is a comfortable, known ceiling, not a surprise. There is no per-user or per-match charge to worry about at our size.

6What this step commits you to

  • Two free homes: GitHub (code + Pages hosting the frontend) and one Supabase project for everything else — likely just one new account, since the code is on GitHub already.
  • One place to publish the app, one place to hold the data — connected only in the browser, at runtime.
  • No servers to manage, patch, or pay for. Both services keep themselves running.
  • Access is by URL for now. At launch the app is reachable by anyone with the address (obscurity + the honour system, as decided). Actually restricting it to the office is the login story — that's Step 5.
